# Tool Deep Dive: Island Analytics

## 1. Scope

Island Analytics is the authenticated workspace analytics surface that starts at `/app` and expands into project-level and report-level views.

- Tool id in hub registry: `island-analytics`. (source: src/tool-hubs/registry.ts:30)
- Hub route target: `/app`. (source: src/tool-hubs/registry.ts:31)
- App route guard: `ProtectedRoute` wraps `/app`. (source: src/App.tsx:116, src/components/ProtectedRoute.tsx:15)

## 2. User Entry Points

### 2.1 Workspace Home

- Route `/app` loads `AppDashboard`. (source: src/App.tsx:117)
- `AppDashboard` reads and creates projects in `projects` table. (source: src/pages/AppDashboard.tsx:46, src/pages/AppDashboard.tsx:84)
- It also computes upload/report counters using `uploads` and `reports`. (source: src/pages/AppDashboard.tsx:52, src/pages/AppDashboard.tsx:53)

### 2.2 Project Drilldown

- Route `/app/projects/:id` loads `ProjectDetail`. (source: src/App.tsx:119)
- `ProjectDetail` loads project metadata and upload history with embedded reports. (source: src/pages/ProjectDetail.tsx:62, src/pages/ProjectDetail.tsx:63)

### 2.3 Report Drilldown

- Route `/app/projects/:id/reports/:reportId` loads `ReportDashboard`. (source: src/App.tsx:120)
- `ReportDashboard` reads `reports` row payload (`parsed_data`, `metrics`, `diagnostics`, `ai_summary`). (source: src/pages/ReportDashboard.tsx:124)

## 3. Frontend Data Flow

## 3.1 Project and Report Persistence

1. User uploads ZIP in `ZipUploader`.
2. ZIP is parsed by `processZipFile`.
3. Metrics are calculated by `calculateMetrics`.
4. Upload metadata is inserted into `uploads`.
5. Report payload is inserted into `reports`.
6. User is redirected to report dashboard.

Evidence:
- ZIP parser and metrics engine calls. (source: src/components/ZipUploader.tsx:17, src/components/ZipUploader.tsx:18, src/components/ZipUploader.tsx:71, src/components/ZipUploader.tsx:83)
- Upload insert flow. (source: src/pages/ProjectDetail.tsx:98)
- Report insert flow with parsed data and metric packs. (source: src/pages/ProjectDetail.tsx:109, src/pages/ProjectDetail.tsx:114, src/pages/ProjectDetail.tsx:115)
- Navigation to report route. (source: src/pages/ProjectDetail.tsx:123)

## 3.2 Parsing and Metrics Subsystem

- Dataset classification and canonicalization are handled by parser registry logic. (source: src/lib/parsing/datasetRegistry.ts:82)
- Derived KPIs/timeseries/rankings/distributions are generated by `calculateMetrics`. (source: src/lib/parsing/metricsEngine.ts:112)
- Time series builders and ranking extraction are in the same engine. (source: src/lib/parsing/metricsEngine.ts:41, src/lib/parsing/metricsEngine.ts:166)

## 3.3 Report Dashboard Rendering

- KPI cards are rendered from `report.metrics.kpis`. (source: src/pages/ReportDashboard.tsx:133, src/pages/ReportDashboard.tsx:209)
- Charts are rendered from `report.metrics.timeseries` using Recharts. (source: src/pages/ReportDashboard.tsx:70, src/pages/ReportDashboard.tsx:134)
- Ranking tables and dataset catalog are rendered from report payload. (source: src/pages/ReportDashboard.tsx:85, src/pages/ReportDashboard.tsx:139)

## 4. Discover Live Relationship (Analytics Public Surface)

Although `island-analytics` route is authenticated, analytics visibility also exists publicly in `/discover`.

- Public discover route: `/discover`. (source: src/App.tsx:104)
- `DiscoverLive` uses `useDiscoverLiveQuery` and renders rails/panels/highlights. (source: src/pages/public/DiscoverLive.tsx:23, src/pages/public/DiscoverLive.tsx:275, src/pages/public/DiscoverLive.tsx:293)
- Panel timeline detail uses `discover-panel-timeline` edge function. (source: src/pages/public/DiscoverLive.tsx:540)

This is important for maintenance because project analytics and public discovery analytics share domain language (panels, ccu, exposure), but are different runtime surfaces.

## 5. Auth and Authorization

- `/app` requires authenticated user. (source: src/App.tsx:116, src/components/ProtectedRoute.tsx:15)
- No separate editor/admin check is enforced for island analytics pages.
  - Admin/editor checks are isolated to `/admin`. (source: src/App.tsx:140, src/components/AdminRoute.tsx:16)

## 6. Data Side Effects

### 6.1 Writes

- `projects` insert/update from dashboard and project page. (source: src/pages/AppDashboard.tsx:84, src/pages/ProjectDetail.tsx:80)
- `uploads` insert on ZIP processing completion. (source: src/pages/ProjectDetail.tsx:98)
- `reports` insert including processed analytics payload. (source: src/pages/ProjectDetail.tsx:109)

### 6.2 Reads

- Project list and metadata reads. (source: src/pages/AppDashboard.tsx:46, src/pages/ProjectDetail.tsx:62)

## 7. Failure Modes

- Invalid ZIP or unsupported structure is surfaced in uploader logs. (source: src/components/ZipUploader.tsx:52, src/components/ZipUploader.tsx:103)
- DB insert failures in upload/report persistence are surfaced through toast and abort flow. (source: src/pages/ProjectDetail.tsx:107, src/pages/ProjectDetail.tsx:119, src/pages/ProjectDetail.tsx:125)
- Report dashboard has no data fallback state when report row is missing. (source: src/pages/ReportDashboard.tsx:176)

## 8. Operational Notes

- This tool family currently persists analytics payloads directly in DB JSON fields (`parsed_data`, `metrics`, `diagnostics`) rather than moving everything through edge functions. (source: src/pages/ProjectDetail.tsx:114, src/pages/ProjectDetail.tsx:115, src/pages/ProjectDetail.tsx:116)
- If migration to edge function orchestration is planned, map this flow to `discover-data-api` style access control first to avoid permission drift. (source: supabase/functions/discover-data-api/index.ts:18, supabase/functions/discover-data-api/index.ts:325)

Evidence:

3. Confirm `reports` row insert succeeded and includes `metrics`.
4. Confirm navigation path points to existing `reportId`.
